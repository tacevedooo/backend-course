Módulo 1

Fundamentos del Backend

¿Qué es el Backend?

El desarrollo backend se ocupa de las tecnologías responsables de almacenar y manipular de forma segura los datos del usuario. Es la parte asociada a toda la lógica oculta que impulsa las aplicaciones con las que interactúan los usuarios.

¿Cuál es la diferencia entre Frontend y Backend?

Tanto el front end como el back end son aspectos críticos de cualquier aplicación. El front end es aquello que ven los usuarios e incluye elementos visuales, como botones, casillas de verificación, gráficos y mensajes de texto. Permite a los usuarios interactuar con la aplicación. El back-end son los datos y la infraestructura que permiten que la aplicación funcione. Almacena y procesa los datos de las aplicaciones para los usuarios.

¿Qué es la arquitectura cliente-servidor?

La arquitectura cliente-servidor es un modelo muy utilizado en el desarrollo de aplicaciones. En este modelo existen dos componentes principales: el cliente, que solicita y utiliza los servicios, y el servidor, que proporciona esos servicios. Ambos se comunican a través de una red, generalmente mediante Internet.

En las aplicaciones web, el cliente suele ser el navegador, mientras que el servidor es el sistema donde se ejecutan las aplicaciones y se almacenan los datos. Actualmente, a la parte del servidor se le conoce como backend, y a la parte que interactúa con el usuario en el cliente se le llama frontend.

El Ciclo de una Petición Web

Todo comienza cuando el cliente realiza una acción (como hacer clic en un enlace). Esto genera una solicitud (Request) que viaja por internet hasta el servidor. El servidor recibe esta solicitud, la procesa (por ejemplo, buscando información en la base de datos) y devuelve una respuesta (Response) al cliente, que finalmente se muestra en pantalla.

¿Qué es el protocolo HTTP/HTTPS?

HTTP (HyperText Transfer Protocol) es el protocolo que permite la comunicación entre el cliente (navegador o aplicación) y el servidor en la web mediante un sistema de solicitudes y respuestas. Sin embargo, los datos se envían sin cifrado, por lo que pueden ser interceptados. HTTPS (HyperText Transfer Protocol Secure) es la versión segura que protege la comunicación mediante cifrado (SSL/TLS), asegurando que la información no pueda ser leída o modificada por externos.

Métodos HTTP

Los métodos indican la intención de la solicitud. Los más utilizados en el backend son:

  • GET: Solicitar o leer datos (Ej. ver un perfil de usuario).
  • POST: Enviar datos para crear un recurso nuevo (Ej. registrarse en una app).
  • PUT: Actualizar por completo un recurso existente.
  • PATCH: Actualizar parcialmente un recurso existente.
  • DELETE: Eliminar un recurso.

Estructura: Headers y Body

Las peticiones y respuestas HTTP tienen dos partes fundamentales: los Headers (cabeceras), que contienen metadatos de configuración (como el tipo de contenido o tokens de seguridad); y el Body (cuerpo), que transporta la carga útil de los datos, como la información de un formulario llenado por el usuario.

Códigos de Estado (Status Codes)

El servidor responde con un número de 3 dígitos que resume el resultado de la petición:

  • 2xx (Éxito): La petición fue procesada correctamente (Ej. 200 OK, 201 Created).
  • 3xx (Redirección): Se requiere una acción adicional, como ir a otra URL.
  • 4xx (Errores del Cliente): Hubo un error en la solicitud (Ej. 400 Bad Request, 404 Not Found).
  • 5xx (Errores del Servidor): El servidor falló al procesar una petición válida (Ej. 500 Internal Server Error).

REST vs RPC

Son los dos enfoques principales para diseñar APIs. REST se enfoca en "recursos" (sustantivos) y usa los métodos HTTP estándar para interactuar con ellos. RPC (Remote Procedure Call) se enfoca en "acciones" (verbos) y consiste en llamar funciones específicas del servidor remotamente, usualmente centralizando todo a través del método POST.

¿Qué es el formato JSON?

JSON (JavaScript Object Notation) es un formato de texto ligero, estructurado y fácil de leer, diseñado para el almacenamiento e intercambio de datos entre sistemas. Es el estándar moderno para enviar información entre un servidor y aplicaciones web.

¿Cómo se comunica el Frontend con el Backend?

Ambos se comunican mediante el protocolo HTTP/HTTPS. El frontend envía solicitudes (requests) y el backend devuelve respuestas, usualmente en formato JSON. Esta interacción estructurada es lo que conocemos como una API (Interfaz de Programación de Aplicaciones).

Preparación del entorno Backend

¿Qué es Node.js?

Durante mucho tiempo, JavaScript solo podía ejecutarse dentro de un navegador web (Frontend). Node.js revolucionó esto al proporcionar un entorno que permite ejecutar JavaScript directamente en el servidor (Backend). Es rápido, altamente escalable y utiliza un modelo "asíncrono", lo que lo hace excelente para construir APIs y manejar miles de conexiones simultáneas.

🎯 Primera tarea: Descarga e instala Node.js en tu computadora. Para verificar que se instaló correctamente, abre tu terminal y ejecuta node -v.

Gestores de paquetes: npm y yarn

En el backend no programamos todo desde cero, aprovechamos librerías creadas por la comunidad. Para gestionarlas usamos gestores de paquetes. npm (Node Package Manager) viene instalado por defecto con Node.js y es el más utilizado, mientras que Yarn es una alternativa popular de alto rendimiento.

Manejo de dependencias y Versionado Semántico

Las librerías externas que usamos se conocen como "dependencias". Para evitar que una actualización externa rompa nuestro código, la industria usa el versionado semántico (SemVer). Es un formato de tres números (por ejemplo: 2.1.4) que representan: versión Mayor (cambios que rompen compatibilidad), versión Menor (nuevas características compatibles) y Parche (correcciones de errores).

🎯 Segunda tarea: Inicializa tu proyecto con nodejs.

Git, GitHub y Flujo de trabajo

Git es un sistema de control de versiones que actúa como una "máquina del tiempo" para tu código, guardando el historial de cada cambio. GitHub es una plataforma en la nube donde alojamos nuestros repositorios de Git para tener respaldos y colaborar con otros desarrolladores.

El flujo de trabajo profesional básico consiste en: crear una rama aislada (branch), guardar nuestros cambios localmente (commit), subirlos a la nube (push) y finalmente solicitar que nuestro código se fusione con el proyecto principal (Pull Request).

🎯 Tercera tarea: Crea una cuenta en GitHub (si aún no la tienes), crea un nuevo repositorio vacío y enlaza tu proyecto local.
