Módulo 4

Autenticación y Seguridad

En el módulo anterior logramos persistir datos reales en una base de datos. Pero hay un gran problema: cualquier persona puede acceder a tus endpoints y hacer lo que quiera con tus datos. En este módulo aprenderemos a proteger nuestra API: quién puede entrar, qué puede ver, y cómo evitar los ataques más comunes.

Helmet: Cabeceras HTTP Seguras

Cuando tu servidor responde una petición, no solo envía datos, también envía cabeceras HTTP (metadatos sobre la respuesta). Por defecto, Express expone información que puede ser aprovechada por atacantes, como qué tecnología usas o qué versión del servidor corre.

helmet es un middleware que configura automáticamente estas cabeceras para seguir las mejores prácticas de seguridad. Es como ponerle un casco a tu servidor.

Instalación
npm install helmet
index.js — Usar Helmet
import helmet from 'helmet';

app.use(helmet()); // Una sola línea protege múltiples vectores de ataque

CORS: Quién Puede Hablarle a tu API

CORS (Cross-Origin Resource Sharing) es una política del navegador que controla qué dominios pueden hacer peticiones a tu API. Por defecto, si tu frontend está en http://localhost:5173 y tu backend en http://localhost:3000, el navegador bloqueará esa comunicación.

El paquete cors te permite definir exactamente qué orígenes están permitidos.

Instalación
npm install cors
index.js — Configurar CORS
import cors from 'cors';

// Opción 1: Permitir TODOS los orígenes (útil en desarrollo)
app.use(cors());

// Opción 2: Permitir solo tu frontend en producción (recomendado)
app.use(cors({
  origin: 'https://mi-frontend.com',
  methods: ['GET', 'POST', 'PUT', 'DELETE']
}));

En producción siempre deberías usar la Opción 2. Abrir CORS a todos los orígenes es como dejar la puerta de tu casa abierta: funciona, pero no es seguro.

Bcrypt: Nunca Guardes Contraseñas en Texto Plano

Una de las reglas más importantes de seguridad: jamás guardes contraseñas como texto legible en tu base de datos. Si alguien hackea tu DB y las contraseñas están en texto plano, todos tus usuarios quedan expuestos.

bcrypt es una función de hashing: transforma una contraseña en una cadena irreversible. No existe forma directa de "desencriptar" un hash de bcrypt; la única manera de verificarlo es hasheando de nuevo la contraseña ingresada y comparando.

Instalación
npm install bcryptjs
index.js — Hashear al guardar (MongoDB / Mongoose)
import bcrypt from 'bcryptjs';

// El hook .pre('save') se ejecuta ANTES de guardar en la DB
userSchema.pre('save', async function() {
  // Solo hashea si la contraseña fue modificada (evita doble hash)
  if (!this.isModified('password')) return;
  
  // El número 10 es el "salt rounds": cuántas veces se procesa
  // Más alto = más seguro, pero más lento. 10 es el estándar.
  this.password = await bcrypt.hash(this.password, 10);
});
index.js — Hashear al guardar (MySQL / mysql2)
const User = sequelize.define('User', {
  name:     { type: DataTypes.STRING, allowNull: false },
  email:    { type: DataTypes.STRING, allowNull: false, unique: true,
              validate: { isEmail: true } },
  password: { type: DataTypes.STRING, allowNull: false }
}, {
  hooks: {
    beforeCreate: async (user) => {
      user.password = await bcrypt.hash(user.password, 10);
    },
    beforeUpdate: async (user) => {
      if (user.changed('password'))
        user.password = await bcrypt.hash(user.password, 10);
    }
  }
});

JWT: Tokens de Identidad

Después del login, necesitamos una forma de identificar al usuario en cada petición siguiente. HTTP es un protocolo sin estado (stateless): el servidor no recuerda quién eres de una petición a otra. Las sesiones tradicionales usaban cookies en el servidor; las APIs modernas usan JWT (JSON Web Tokens).

Un JWT es una cadena firmada criptográficamente que contiene información del usuario. El servidor la genera al hacer login y el cliente la envía en cada petición siguiente.

Instalación
npm install jsonwebtoken
index.js — Generar y verificar JWT
import jwt from 'jsonwebtoken';

// Esta clave NUNCA debe estar en el código
// Guárdala en una variable de entorno (.env)
const JWT_SECRET = 'mi_clave_secreta_123';

// GENERAR TOKEN al hacer login exitoso
app.post('/login', async (req, res) => {
  // ... validar credenciales ...
  
  // jwt.sign(payload, secreto, opciones)
  const token = jwt.sign(
    { id: user._id },      // Payload: qué datos incluir (nunca la contraseña)
    JWT_SECRET,
    { expiresIn: '1h' }    // El token expira en 1 hora
  );
  
  res.json({ token });
});

// VERIFICAR TOKEN en un middleware
const authMiddleware = (req, res, next) => {
  // El cliente envía el token en el header: "Authorization: Bearer "
  const authHeader = req.headers.authorization;
  const token = authHeader && authHeader.split(' ')[1]; // Extrae solo el token

  if (!token) return res.status(401).json({ error: "Token no proporcionado" });

  try {
    // jwt.verify() lanza un error si el token es inválido o expiró
    const decoded = jwt.verify(token, JWT_SECRET);
    req.user = decoded; // Adjunta los datos del usuario a la request
    next();             // Continúa al siguiente handler
  } catch (error) {
    res.status(401).json({ error: "Token inválido o expirado" });
  }
};
index.js — Proteger rutas con el middleware
// Ruta PÚBLICA: cualquiera puede acceder
app.post('/login', async (req, res) => { ... });
app.post('/register', async (req, res) => { ... });

// Ruta PROTEGIDA: solo usuarios con token válido
// authMiddleware se ejecuta ANTES de la ruta /users
app.get('/users', authMiddleware, async (req, res) => {
  // Si llegamos aquí, req.user existe y tiene el id del usuario
  const users = //lógica para obtener usuarios de la DB;
  res.json(users);
});

Flujo completo de autenticación

Servidor Completo: MongoDB + Seguridad

Aquí está el servidor completo con todas las piezas ensambladas usando MongoDB y Mongoose. Cada sección está comentada para que puedas identificar exactamente qué hace cada parte.

index.js — Servidor MongoDB con autenticación
import express from 'express';
import mongoose from 'mongoose';
import bcrypt from 'bcryptjs';
import jwt from 'jsonwebtoken';
import cors from 'cors';
import helmet from 'helmet';

const app = express();
const PORT = 3000;
const JWT_SECRET = 'mi_clave_secreta_123'; 

// 1. SEGURIDAD BÁSICA
app.use(helmet()); 
app.use(cors());
app.use(express.json());

// 2. CONEXIÓN A MONGODB
mongoose.connect('mongodb://localhost:27017/test')
  .then(() => console.log("Conectado a MongoDB"))
  .catch(err => console.error("Error de conexión:", err));

// 3. MODELO con hook de encriptación
const userSchema = new mongoose.Schema({
  name:     { type: String, required: true },
  email:    { type: String, required: true, unique: true },
  password: { type: String, required: true }
});

userSchema.pre('save', async function() {
  if (!this.isModified('password')) return;
  this.password = await bcrypt.hash(this.password, 10);
});

const User = mongoose.model('User', userSchema);

// 4. MIDDLEWARE JWT
const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: "Token no proporcionado" });
  try {
    req.user = jwt.verify(token, JWT_SECRET);
    next();
  } catch {
    res.status(401).json({ error: "Token inválido o expirado" });
  }
};

// 5. RUTAS PÚBLICAS
app.post('/register', async (req, res) => {
  try {
    const { name, email, password } = req.body;
    if (!name || !email || !password)
      return res.status(400).json({ error: "Todos los campos son obligatorios" });

    const newUser = new User({ name, email, password });
    await newUser.save();
    res.status(201).json({ message: "Usuario registrado", id: newUser._id });
  } catch (error) {
    res.status(400).json({ error: "El email ya existe o datos inválidos" });
  }
});

app.post('/login', async (req, res) => {
  try {
    const { email, password } = req.body;
    const user = await User.findOne({ email });

    if (user && await bcrypt.compare(password, user.password)) {
      const token = jwt.sign({ id: user._id }, JWT_SECRET, { expiresIn: '1h' });
      res.json({ token });
    } else {
      res.status(401).json({ error: "Credenciales inválidas" });
    }
  } catch {
    res.status(500).json({ error: "Error en el servidor" });
  }
});

// 6. RUTAS PROTEGIDAS
app.get('/users', authMiddleware, async (req, res) => {
  const users = await User.find().select('-password');
  res.json(users);
});

app.put('/users/:id', authMiddleware, async (req, res) => {
  const updated = await User.findByIdAndUpdate(
    req.params.id, { name: req.body.name }, { new: true }
  ).select('-password');
  if (!updated) return res.status(404).json({ message: "No encontrado" });
  res.json(updated);
});

app.delete('/users/:id', authMiddleware, async (req, res) => {
  const deleted = await User.findByIdAndDelete(req.params.id);
  if (!deleted) return res.status(404).json({ message: "No encontrado" });
  res.json({ message: "Usuario eliminado" });
});

app.listen(PORT, () => console.log(`Servidor listo en http://localhost:${PORT}`));

Servidor Completo: MySQL + Sequelize + Seguridad

La misma lógica de autenticación aplicada a MySQL. La diferencia principal es que usamos Sequelize como ORM (Object-Relational Mapper), que hace para SQL lo que Mongoose hace para MongoDB: nos permite trabajar con objetos JS en lugar de escribir SQL puro.

Instalación (MySQL)
npm install sequelize mysql2 bcryptjs jsonwebtoken cors helmet
index.js — Servidor MySQL con Sequelize y autenticación
import express from 'express';
import { Sequelize, DataTypes } from 'sequelize';
import bcrypt from 'bcryptjs';
import jwt from 'jsonwebtoken';
import cors from 'cors';
import helmet from 'helmet';

const app = express();
const PORT = 3000;
const JWT_SECRET = 'mi_clave_secreta_123';

// 1. SEGURIDAD BÁSICA
app.use(helmet());
app.use(cors());
app.use(express.json());

// 2. CONEXIÓN A MYSQL (Sequelize)
const sequelize = new Sequelize('test_db', 'root', 'tu_password', {
  host: 'localhost',
  dialect: 'mysql',
  logging: false
});

// 3. MODELO con hooks de encriptación
const User = sequelize.define('User', {
  name:     { type: DataTypes.STRING, allowNull: false },
  email:    { type: DataTypes.STRING, allowNull: false, unique: true,
              validate: { isEmail: true } },
  password: { type: DataTypes.STRING, allowNull: false }
}, {
  hooks: {
    beforeCreate: async (user) => {
      user.password = await bcrypt.hash(user.password, 10);
    },
    beforeUpdate: async (user) => {
      if (user.changed('password'))
        user.password = await bcrypt.hash(user.password, 10);
    }
  }
});

// Crea la tabla si no existe
sequelize.sync()
  .then(() => console.log("Conectado a MySQL y tabla sincronizada"))
  .catch(err => console.error("Error:", err));

// 4. MIDDLEWARE JWT (idéntico al de MongoDB)
const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: "Token no proporcionado" });
  try {
    req.user = jwt.verify(token, JWT_SECRET);
    next();
  } catch {
    res.status(401).json({ error: "Token inválido o expirado" });
  }
};

// 5. RUTAS PÚBLICAS
app.post('/register', async (req, res) => {
  try {
    const { name, email, password } = req.body;
    const newUser = await User.create({ name, email, password });
    res.status(201).json({ message: "Usuario registrado", id: newUser.id });
  } catch {
    res.status(400).json({ error: "El email ya existe o datos inválidos" });
  }
});

app.post('/login', async (req, res) => {
  try {
    const { email, password } = req.body;
    const user = await User.findOne({ where: { email } });

    if (user && await bcrypt.compare(password, user.password)) {
      const token = jwt.sign({ id: user.id }, JWT_SECRET, { expiresIn: '1h' });
      res.json({ token });
    } else {
      res.status(401).json({ error: "Credenciales inválidas" });
    }
  } catch {
    res.status(500).json({ error: "Error en el servidor" });
  }
});

// 6. RUTAS PROTEGIDAS
app.get('/users', authMiddleware, async (req, res) => {
  const users = await User.findAll({ attributes: { exclude: ['password'] } });
  res.json(users);
});

app.put('/users/:id', authMiddleware, async (req, res) => {
  const user = await User.findByPk(req.params.id);
  if (!user) return res.status(404).json({ message: "No encontrado" });
  await user.update({ name: req.body.name });
  res.json({ id: user.id, name: user.name, email: user.email });
});

app.delete('/users/:id', authMiddleware, async (req, res) => {
  const deleted = await User.destroy({ where: { id: req.params.id } });
  if (!deleted) return res.status(404).json({ message: "No encontrado" });
  res.json({ message: "Usuario eliminado" });
});

app.listen(PORT, () => console.log(`Servidor MySQL listo en http://localhost:${PORT}`));

Actividad Práctica: API con Autenticación y Seguridad

Ahora que entiendes cada pieza por separado, es momento de ensamblarlas. Toma tu proyecto del Módulo 3 (el CRUD con base de datos) y agrégale la capa de autenticación y seguridad.

🎯 Tarea del Módulo: Implementa autenticación y seguridad en tu servidor (MongoDB o MySQL). Debe incluir registro con contraseña hasheada, login con generación de JWT, y rutas CRUD protegidas por el middleware de autenticación. Las credenciales sensibles deben estar en variables de entorno. Recuerda Testear cada endpoint con Postman para verificar que la autenticación funciona correctamente. Puedes ayudarte de la inteligencia artificial.

✅ Checklist del Módulo 4

Antes de avanzar, asegúrate de:

  • 1. Helmet y CORS: Tu servidor usa helmet() y cors() como middlewares globales.
  • 2. Contraseñas hasheadas: Ninguna contraseña se guarda en texto plano; todas pasan por bcrypt antes de llegar a la DB.
  • 3. JWT funcional: El endpoint /login genera un token y el authMiddleware lo verifica correctamente.
  • 4. Rutas protegidas: Las rutas CRUD devuelven 401 si no se envía un token válido.
  • 5. Variables de entorno: El JWT_SECRET y las credenciales de DB están en un archivo .env que no se sube al repositorio.
