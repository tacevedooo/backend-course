Módulo 5

Arquitectura Backend Profesional

En el módulo anterior logramos asegurar nuestra API con autenticación, JWT y contraseñas encriptadas. Pero seguramente notaste algo que tu archivo index.js se está volviendo enorme. Tienes la configuración del servidor, la conexión a la base de datos, las rutas, la lógica de negocio y la seguridad, todo mezclado en un solo lugar.

En la industria, el código no se escribe así. En este módulo aprenderemos sobre Modularidad y Arquitectura: Dividir un problema grande en piezas pequeñas, ordenadas y fáciles de mantener.

¿Qué es la Modularidad?

La modularidad en la arquitectura de software es simplemente la práctica de dividir cosas grandes en piezas más pequeñas.

  • Mejor organización: Te ayuda a visualizar y entender qué hace cada parte de tu proyecto.
  • Menos errores: Un proyecto subdividido es mucho más fácil de mantener y probar.
  • Trabajo en equipo: Si tu proyecto tiene muchas piezas, diferentes programadores pueden trabajar en distintas partes al mismo tiempo sin "pisarse" el código.

Infraestructura: Monolitos vs Microservicios

Antes de ver cómo organizar nuestras carpetas, hablemos de cómo se organizan los servidores en la vida real. Imagina que estamos construyendo una app de streaming de video.

La Arquitectura Monolítica

Al principio, la APP tiene un solo servidor (un backend en Node/Express) que se encarga de todo: autenticar usuarios, procesar los pagos de suscripción y transmitir los videos. A esto se le llama Monolito. Es muy bueno para empezar porque es simple de crear y desplegar.

La Arquitectura de Microservicios

De repente, la APP se vuelve mundialmente famosa. El servidor empieza a colapsar porque transmitir video consume demasiados recursos. En lugar de tener un solo servidor gigante, dividimos el backend: creamos un servidor solo para pagos, otro solo para autenticación y otro solo para streaming. Esto es un Microservicio. Si el streaming necesita más potencia, solo escalamos ese servidor sin tocar los demás.

Nota: Los microservicios son complejos y se usan en proyectos gigantes. Para tu proyecto actual, seguiremos usando un Monolito, pero organizaremos su código internamente como profesionales.

El Problema: Todo en un solo archivo

Imagina que creamos una API para gestionar un refugio de conejitos. Si ponemos todo en app.js, se vería así:

app.js — (Cómo NO hacerlo en proyectos grandes)
const express = require('express');
const app = express();

// 1. Base de datos simulada aquí mismo
const db = [{ id: 1, name: 'Bugs' }, { id: 2, name: 'Lola' }];

// 2. Rutas y Lógica de negocio mezcladas
app.get('/rabbits', (req, res) => {
    // Si la lógica fuera compleja, este archivo tendría miles de líneas
    res.json(db); 
});

app.listen(3000, () => console.log('Servidor listo'));

Esto funciona, pero si agregamos validaciones, seguridad, manejo de errores y conexión a una base de datos real, este archivo sería muy difícil de entender.

Arquitectura por Capas

Para solucionar esto, dividiremos las responsabilidades en diferentes carpetas y archivos. La regla más importante es que cada archivo debe hacer una sola cosa, y la comunicación sigue un flujo estricto: la petición entra por el servidor, pasa a las rutas, luego al controlador y finalmente al modelo.

Nuestra nueva estructura de carpetas será así:



📦 mi-proyecto/
 ┣ 📂 db/
 ┃ ┗ 📜 connection.js      // Solo se conecta a la base de datos
 ┣ 📂 routes/
 ┃ ┗ 📜 users.routes.js    // Solo define las URLs (ej: GET /users)
 ┣ 📂 controllers/
 ┃ ┗ 📜 users.ctrl.js      // El cerebro: recibe la petición y decide qué hacer
 ┣ 📂 models/
 ┃ ┗ 📜 users.model.js     // Solo habla con la Base de Datos (SQL o MongoDB)
 ┗ 📜 app.js               // Solo levanta el servidor de Express


Paso 1: El Servidor (app.js)

Su única responsabilidad es iniciar Express y conectar las rutas principales.

app.js
import express from 'express';
import rabbitRoutes from './routes/rabbits.routes.js';

const app = express();

app.use(express.json());
app.use('/rabbits', rabbitRoutes); // Conectamos la capa de rutas

app.listen(7070, () => console.log('Servidor en puerto 7070'));


Paso 2: Las Rutas (routes/)

Aquí definimos los "caminos" de nuestra API, pero no escribimos la lógica. Solo delegamos el trabajo al controlador.

routes/rabbits.routes.js
import { Router } from 'express';
import { listRabbits, getRabbit } from '../controllers/rabbits.controllers.js';

const router = Router();

router.get('/', listRabbits);    // Delega al controlador
router.get('/:id', getRabbit);   // Delega al controlador

export default router;


Paso 3: Los Controladores (controllers/)

Es el cerebro. Valida los datos de la petición (req) y decide qué responder (res).

controllers/rabbits.controllers.js
import { getAllItems } from '../models/rabbits.models.js';

export const listRabbits = async (req, res) => {
    try {
        const rabbits = await getAllItems(); // Pide datos al modelo
        res.status(200).json(rabbits);
    } catch (error) {
        res.status(500).json({ error: "Error en el servidor" });
    }
};


Paso 4: Los Modelos (models/)

Esta capa es la única autorizada para hablar con la base de datos (MongoDB o MySQL). Si un día cambias de base de datos, solo modificarías esta carpeta!

models/rabbits.models.js
import db from '../db/connection.js'; // Tu conexión a DB

export const getAllItems = async () => {
    // Aquí iría tu código de Mongoose o Sequelize
    // Ej: return await Rabbit.find();
    return [{ id: 1, name: "Bugs" }]; 
};

✅ Checklist del Módulo 5

Antes de avanzar, asegúrate de:

  • 1. Limpieza del servidor: Tu archivo app.js o index.js solo contiene la configuración de Express, middlewares globales y la importación de rutas.
  • 2. Capa de Rutas: Tienes una carpeta routes/ donde se definen los endpoints (ej. /login, /users/:id) y se aplican los middlewares de autenticación.
  • 3. Capa de Controladores: Tienes una carpeta controllers/ que recibe los objetos req y res, ejecuta la lógica en try/catch y devuelve las respuestas HTTP correspondientes.
  • 4. Capa de Modelos: Tienes una carpeta models/ donde están tus esquemas de Mongoose o modelos de Sequelize, siendo el único lugar que interactúa directamente con la DB.
  • 5. Funcionamiento intacto: Has probado tu API refactorizada con Postman y el CRUD + Login sigue funcionando exactamente igual que en el Módulo 4.
