Módulo 6

Despliegue y Producción

En el módulo anterior logramos organizar nuestro código usando la Arquitectura por Capas. Tu API ya es segura, modular y fácil de mantener. Pero hasta ahora, todo vive en nuestro computador (localhost). Si apagas tu PC, tu aplicación deja de existir.

En este módulo aprenderemos sobre Despliegue y Producción: El proceso de tomar tu código fuente, prepararlo y alojarlo en un servidor real en la nube para que esté disponible las 24 horas del día, los 7 días de la semana.

1. Variables de Entorno

La regla número uno de la producción es nunca subir información sensible a tu repositorio de GitHub. Cosas como contraseñas, tu URL de conexión a la base de datos o tus secretos de JWT jamás deben estar en tu código.

Para resolver esto usamos Variables de Entorno. En Node.js, utilizamos una librería llamada dotenv para leer un archivo oculto llamado .env.

.env — (Este archivo JAMÁS se sube a GitHub, ponlo en tu .gitignore)
PORT=8080
DB_URI=mongodb+srv://usuario:passwordSuperSecreta@cluster.mongodb.net/mi_db
JWT_SECRET=MiSuperSecretoInhackeable123
db/connection.js — (Cómo leer las variables)
import 'dotenv/config'; // 1. Carga las variables del archivo .env
import express from 'express';
// import { connectDB } from './db/connection.js'; // Importa tu conexión

const app = express();

// 2. Usa el puerto del .env (producción) o el 3000 por defecto (local)
const PORT = process.env.PORT || 3000; 

app.use(express.json());

// 3. Ejemplo genérico de conexión a tu Base de Datos (SQL o MongoDB)
// connectDB(process.env.DB_URI)
//     .then(() => console.log('Base de datos conectada con éxito'))
//     .catch(error => console.error('Error de conexión:', error));

app.listen(PORT, () => {
    console.log(`Servidor corriendo en el puerto ${PORT}`);
});

2. Bases de Datos en la Nube

Si tu base de datos vive localmente en tu computadora, tu servidor en la nube no podrá acceder a ella. Antes de hacer deploy, necesitas migrar tu base de datos a un servicio gestionado en la nube.

  • Si usas MongoDB: La mejor opción gratuita para empezar es MongoDB Atlas. Te dará una URL de conexión que pondrás en tu archivo .env.
    Ver Tutorial: Cómo configurar MongoDB Atlas
  • Si usas SQL (Postgres/MySQL): Plataformas como Neon, Supabase o Railway ofrecen bases de datos gratuitas y fáciles de configurar.
    Ver Tutorial: Cómo crear un servicio de base de datos en Railway

3. Plataformas como Servicio (PaaS)

Hace años, hacer deploy implicaba rentar una computadora vacía (VPS), instalar Linux, Node.js, configurar firewalls y dominios a mano. Hoy en día, usamos PaaS (Platform as a Service).

Herramientas como Render o Railway se conectan directamente a tu repositorio de GitHub. Cada vez que haces un git push, ellos detectan que es una app de Node.js, instalan las dependencias automáticamente y encienden tu servidor.

Verificar que tu package.json tenga este script
"scripts": {
    "start": "node app.js", // Las plataformas en la nube ejecutarán este comando
}

Actividad Práctica: ¡Sube tu API a Internet!

✅ Checklist del Módulo 6

Antes de avanzar, asegúrate de:

  • 1. Seguridad de credenciales: Tienes un archivo .env para tu entorno local y este archivo NO aparece en tu repositorio de GitHub.
  • 2. Script de inicio: Tu package.json tiene un comando "start": "node app.js" configurado correctamente.
  • 3. Puerto dinámico: Tu servidor escucha en process.env.PORT || 3000, permitiendo que el proveedor de la nube asigne el puerto que necesite.
  • 4. Base de datos en la nube: Tu aplicación está conectada a una base de datos en la nube (como MongoDB Atlas o Railway), no a una local.
  • 5. Backend en la nube: Tienes una URL pública (ej. tu-api.onrender.com) y puedes hacer peticiones exitosas (GET, POST) desde Postman apuntando a esa URL de internet, no a localhost.
